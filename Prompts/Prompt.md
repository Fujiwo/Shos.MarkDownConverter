# Shos.MarkDownConverter 実装プロンプト for GitHub Copilot

あなたは GitHub Copilot として、Shos.MarkDownConverter を実装してください。
まず既存のリポジトリ構成を確認し、現状に合わせて最小限の変更で進めてください。推測で大きく作り替えず、段階的に実装し、各段階で動作確認できる状態を保ってください。

## 1. 目的
Shos.MarkDownConverter は、Microsoft の MarkItDown が対応する各種ファイルを Markdown に変換する Web アプリケーションです。

このアプリケーションで必ず実現すること:

1. ブラウザーから単一ファイルをアップロードできること
2. サーバー側で Python 版 MarkItDown を呼び出して Markdown に変換できること
3. 変換結果を画面に表示できること
4. 変換結果をコピーまたはダウンロードできること

変換対象のファイル形式は、MarkItDown の対応範囲に準拠してください。

## 2. Copilot への作業方針
以下の方針で作業してください。

1. いきなり実装を始めず、まず現在のファイル構成と既存コードを確認すること
2. 不足しているファイルのみを追加し、不要な生成物は作らないこと
3. 過剰な設計や抽象化を避け、まず動く最小構成を完成させること
4. MarkItDown の変換ロジックは再実装せず、外部ツールとして利用すること
5. セキュリティ、エラー処理、テスト、ドキュメントを後回しにしないこと
6. 不明点があっても勝手に機能追加せず、要求範囲内で完結させること

## 3. 技術制約
以下を前提としてください。

### サーバーサイド
- .NET 10
- ASP.NET Core
- .NET 10 で利用できる最新の C#
- データベースは使用しない

### クライアントサイド
- HTML
- CSS
- JavaScript

### 外部連携
- Python 版 MarkItDown を利用する
- ASP.NET Core から外部プロセスとして安全に呼び出す

## 4. 必須機能
最低限、以下を実装してください。

1. 単一ファイルアップロード
2. 入力ファイルのバリデーション
3. MarkItDown 実行処理
4. Markdown 変換結果の表示
5. 結果のコピー
6. 結果のダウンロード
7. 失敗時のユーザー向けエラー表示

追加してよい機能:

1. ドラッグアンドドロップ
2. 対応拡張子一覧の表示
3. 変換中表示
4. ファイルサイズ上限の設定

追加機能より、基本機能の安定性を優先してください。

## 5. 責務分離
責務は次のように分離してください。

### MarkItDown の責務
- 入力ファイルを Markdown に変換すること

### Web アプリケーションの責務
- ファイル受付
- 入力検証
- 外部プロセス実行
- 実行結果の解釈
- UI 表示
- エラー表示
- ログ出力
- 一時ファイル管理
- 設定管理

MarkItDown の変換ルールや変換処理を C# 側で再実装してはいけません。

## 6. アーキテクチャ要件
以下を満たしてください。

1. 単純で理解しやすい構成にすること
2. UI、アプリケーション処理、MarkItDown 呼び出し処理を分離すること
3. 依存関係を最小限に保つこと
4. 将来の設定追加や拡張に耐えられる程度の分離は行うこと
5. ただし、不要なレイヤーやインターフェイスの乱立は避けること

## 7. Python 版 MarkItDown 連携要件
以下を守って実装してください。

1. Python 実行パスを設定可能にすること
2. MarkItDown 呼び出しロジックを一箇所に集約すること
3. 標準出力、標準エラー出力、終了コードを取得すること
4. ユーザー向けエラーと開発者向けログを分けること
5. コマンド文字列連結ではなく、安全な引数指定でプロセス実行すること
6. 一時ファイルや作業ディレクトリを確実にクリーンアップすること
7. Python または MarkItDown が利用できない場合、明確なエラーを返すこと

## 8. セキュリティ要件
最低限、以下を満たしてください。

1. アップロードファイル名を信用しないこと
2. 許可対象外の形式を拒否できること
3. 必要に応じてファイルサイズを制限すること
4. パス操作やコマンドインジェクションを防ぐこと
5. 一時保存先と削除処理を明示すること
6. 例外詳細をそのまま画面に出しすぎないこと

## 9. UI 要件
UI は簡素でよいですが、以下は満たしてください。

1. 初見でも使い方が分かること
2. アップロード操作が分かりやすいこと
3. Markdown 結果が読みやすいこと
4. エラー時に次の確認ポイントが分かること
5. コピーとダウンロード操作が明確であること

## 10. テスト要件
以下のテストを追加してください。

### 単体テスト
- 入力検証
- 対応形式判定
- MarkItDown 呼び出しラッパーの正常系
- MarkItDown 呼び出しラッパーの異常系
- エラー整形処理

### 結合テスト
- ファイルアップロードから結果表示までの流れ
- 変換失敗時の表示
- Python または MarkItDown 不在時の挙動

正常系だけでなく異常系も必ず含めてください。

## 12. Agent Skills と Copilot 設定方針
Microsoft Agent Skills の .NET Skills は導入候補として検討すること。

- Microsoft Agent Skills の .NET Skills を検討する場合は、https://github.com/dotnet/skills のものを候補とすること
- その他 Python や Web 開発などの Agent Skills も、必要性が明確な場合のみ検討すること

ただし、以下の条件を守ってください:

1. 導入自体を目的にしないこと
2. Web アプリ本体に明確な価値がある場合のみ採用すること
3. 不要であれば導入しないこと
4. 導入する場合は、採用理由と利用箇所を説明できるようにすること

copilot-instructions.md を作成または更新する場合は、以下を含めてください。

1. プロジェクトの目的
2. 採用技術
3. 実装方針
4. テスト方針
5. MarkItDown 連携時の注意点
6. セキュリティ上の注意点

## 13. 禁止事項
以下は行わないでください。

1. データベースの導入
2. MarkItDown の変換処理の再実装
3. 必要性の薄い依存関係の追加
4. 過剰なレイヤー分割
5. UI だけを先に作り込み、バックエンドを後回しにすること
6. 例外を握りつぶすこと
7. 設定値のハードコードを乱用すること

1. まず現状を確認してから変更すること
2. 変更理由が説明できる実装にすること
3. 変更範囲を必要最小限に保つこと
